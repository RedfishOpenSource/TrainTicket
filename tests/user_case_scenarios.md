# 火车票最优购买功能测试场景

本文档用于沉淀“火车票最优购买功能”的业务测试场景，便于后续持续迭代时做回归核对。

## 测试目标

给定输入：
- 出发日期
- 出发站
- 到达站
- 可选：指定车次

系统需要输出：
- 当前可购买的最优方案
- 明确的购票步骤
- 总耗时、总票价、方案类型
- 只要存在现票，换乘方案也属于合法方案

排序规则：
1. 优先按实际乘坐区间的总耗时最短排序
2. 总耗时相同，再按总票价最低排序
3. 如仍相同，可按分段数更少优先

---

## 方案类型定义

- `direct`：直接购买目标区间
- `buy_longer`：购买同车次更长区间，实际乘坐目标区间
- `split_ticket`：同车次拆段购买
- `transfer`：不同车次换乘购买
- `none`：无可行现票方案

---

## 核心业务场景

### Case 1：直达有票
**输入**
- 日期：任意有效日期
- 出发站：A
- 到达站：B

**前置条件**
- 同车次目标区间 `A -> B` 有现票

**期望结果**
- 返回 `direct`
- 不返回更复杂的 `buy_longer` / `split_ticket` / `transfer` 作为最优方案
- 总耗时按 `A -> B` 实际乘坐时间计算
- 总票价为目标区间票价

---

### Case 2：直达无票，但同车次买长坐短可行
**输入**
- 日期：例如 `2026-05-01`
- 出发站：西安
- 到达站：十堰
- 可选指定车次：`K2098`

**前置条件**
- `西安 -> 十堰` 无现票
- 同一物理列车上存在覆盖区间票，例如：
  - `大同 -> 十堰`
  - `大同 -> 长沙`
  - `河津 -> 十堰`
- 用户可以在西安上车、十堰下车

**期望结果**
- 返回 `buy_longer`
- 方案中明确写出“购买哪个更长区间”
- 购票步骤中明确写出“实际乘坐区间为 西安 -> 十堰”
- 总耗时按 `西安 -> 十堰` 实际乘坐时间计算，不按买票区间计算

---

### Case 3：直达无票，但同车次拆段可行
**输入**
- 日期：任意有效日期
- 出发站：A
- 到达站：C

**前置条件**
- `A -> C` 无现票
- `A -> B` 有票
- `B -> C` 有票
- 两段属于同一趟车

**期望结果**
- 返回 `split_ticket`
- 明确列出每一段购票步骤
- 总票价为各段票价之和
- 总耗时按 `A -> C` 实际乘坐时间计算

---

### Case 4：只有后续区间有票，不满足原始上车站要求
**输入**
- 日期：例如 `2026-05-01`
- 出发站：西安
- 到达站：十堰
- 可选指定车次：`K2098`

**前置条件**
- `西安 -> 十堰` 无票
- `安康 -> 十堰` 有票
- 但用户的出发站是 `西安`

**期望结果**
- 不能把 `安康 -> 十堰` 误判为满足 `西安 -> 十堰`
- 如果没有覆盖 `西安 -> 十堰` 的合法买长坐短或拆段方案，则应返回 `none`
- 页面或结果层不能把“后续区间有票”直接当成用户可执行方案

---

### Case 5：同车次存在多个可行方案时按统一规则选最优
**输入**
- 日期：任意有效日期
- 出发站：A
- 到达站：C

**前置条件**
- 同时存在：
  - `buy_longer`
  - `split_ticket`
  - 或多个不同的 `buy_longer`

**期望结果**
- 优先比较实际乘坐区间总耗时
- 总耗时相同，再比较总票价
- 再相同，则分段数更少优先
- 返回的最优方案必须符合统一排序规则

---

### Case 6：同车次不可行，但跨车次换乘可行
**输入**
- 日期：任意有效日期
- 出发站：A
- 到达站：C

**前置条件**
- 同车次 `direct` / `buy_longer` / `split_ticket` 都不可行
- 不同车次换乘可行，例如：
  - `A -> B` 有票
  - `B -> C` 有票
  - 且换乘时间满足约束

**期望结果**
- 返回 `transfer`
- 只要换乘后的各段都有现票，该方案就是合法方案
- 不要求必须同车次，只要能买到票并最终到达目的地即可
- 列出每段车次、上车站、下车站、座席、价格
- 总耗时按首次发车到最终到达计算
- 总票价为所有换乘段票价之和

---

### Case 7：完全无票
**输入**
- 日期：任意有效日期
- 出发站：A
- 到达站：B

**前置条件**
- `direct` / `buy_longer` / `split_ticket` / `transfer` 均无现票

**期望结果**
- 返回空方案或 `none`
- 不得把候补、无票、`--` 识别为现票

---

## 12306 特殊规则场景

### Case 8：跨天列车只要在用户指定日期到达上车站，就满足该日出发需求
**典型示例**
- 用户需求：`5.1` 从 `西安` 出发到 `十堰`
- 同一列车从 `大同` 始发
- 该列车在 `4.30` 从 `大同` 发车
- 该列车在 `5.1` 到达 `西安`，随后继续开往 `十堰`

**前置条件**
- 列车始发日期早于用户指定日期
- 但列车在用户指定日期到达用户上车站
- 用户可以在该日于上车站正常上车

**期望结果**
- 该列车应被视为满足“`5.1` 从 `西安` 出发到 `十堰`”的需求
- 系统在补同车次覆盖区间或拆段区间时，应围绕用户实际上车日期进行判断
- 不能因为列车是 `4.30` 从始发站发车，就错误排除这趟车
- 若存在如 `大同 -> 十堰`、`大同 -> 长沙`、`河津 -> 十堰` 这类覆盖 `西安 -> 十堰` 的现票方案，应视为合法候选

---

### Case 9：候补不等于现票
**前置条件**
- 接口返回值为“候补”或等价状态

**期望结果**
- 不算作可购买方案
- 不进入最优方案候选

---

### Case 10：无座属于合法现票
**前置条件**
- 区间存在 `无座` 现票

**期望结果**
- `无座` 作为合法座席类型参与比较
- 可以成为最优方案

---

### Case 11：多座席并存时应保留全部可售选项
**前置条件**
- 同一区间存在 `无座 / 硬座 / 硬卧` 等多个现票座席

**期望结果**
- 结果中保留全部可售座席信息
- 若要取单方案，默认选择该区间最便宜的可售座席

---

### Case 12：同一物理列车中途显示车次变化
**典型示例**
- 同一趟物理车列在前段显示为 `K2095`
- 到西安及后续区段显示为 `K2098`
- 但其 `train_no` 不变

**期望结果**
- 补同车次覆盖段时，应按稳定标识匹配同一物理列车
- 不能因显示车次变化而漏掉覆盖购票方案

---

## 当前重点业务场景（K2098）

### Case K1：目标区间无票，但存在更早上车的覆盖票
**输入**
- 日期：`2026-05-01`
- 出发站：`西安`
- 到达站：`十堰`
- 车次：`K2098`

**前置条件**
- `西安 -> 十堰` 无票
- 存在覆盖区间现票，例如：
  - `河津 -> 十堰`
  - `大同 -> 长沙`
  - `太原 -> 长沙`

**期望结果**
- 程序应识别出这些覆盖区间属于同一物理列车
- 输出 `buy_longer` 类型最优方案
- 方案中的总耗时按 `西安 -> 十堰` 计算
- 总票价按买到的覆盖区间票价计算

---

### Case K2：虽然后续站有票，但不能误算成西安出发方案
**输入**
- 日期：`2026-05-01`
- 出发站：`西安`
- 到达站：`十堰`
- 车次：`K2098`

**前置条件**
- 仅 `安康 -> 十堰` 有票
- `西安` 无法合法上车覆盖到该票方案

**期望结果**
- 不能直接把 `安康 -> 十堰` 当作 `西安 -> 十堰` 的可执行方案

---

### Case K3：多个覆盖区间同时可买时，输出最优一个
**输入**
- 日期：`2026-05-01`
- 出发站：`西安`
- 到达站：`十堰`
- 车次：`K2098`

**前置条件**
- 存在多个覆盖区间票，例如：
  - `河津 -> 十堰 无座`
  - `大同 -> 长沙 硬座`
  - `太原 -> 长沙 硬座`

**期望结果**
- 这些方案都应被识别为可行候选
- 最终按统一排序规则选出最优一个

---

## 结果展示要求

每个有效方案至少应展示：
- 方案类型
- 总耗时
- 总票价
- 购票步骤
- 每段车次
- 每段上车站 / 下车站
- 每段座席类型
- 必要提示（如“实际乘坐区间为 …”）

---

## 回归建议

后续每次迭代建议至少人工核对以下场景：
- 直达有票
- 买长坐短
- 同车次拆段
- 后续区间有票但不合法
- 换乘（只要有票就属于合法候选）
- 跨天
- 候补过滤
- 无座可售
- 同车改号
- K2098 实战场景
