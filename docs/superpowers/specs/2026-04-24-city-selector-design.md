# 城市维度选择器设计

## 背景
当前页面中的出发/到达选择器本质上仍是基于 12306 站点的搜索下拉，用户需要直接面对具体站名。对于北京、上海等多站城市，这会让用户在输入阶段就被迫理解站点颗粒度，使用门槛偏高。

本次调整将把选择入口升级为城市维度选择器，但保留最终查询仍基于具体 12306 站点，从而在不改动核心购票方案计算链路的前提下，提升选择体验。

## 目标
- 用户按城市维度搜索出发地和目的地
- 当城市只对应一个站点时，用户可直接完成选择
- 当城市对应多个站点时，用户必须在组件内确认具体站点
- 表单最终提交的仍是有效的 12306 站点名
- 现有查询、推荐、结果展示逻辑保持不变

## 非目标
- 不改动购票方案计算逻辑
- 不在首版引入热门城市、最近使用城市
- 不引入省份级联或地图式选择
- 不替换成重量级前端框架组件库

## 总体方案
采用“城市搜索 + 站点确认”的两阶段选择模型。

前端将当前站点搜索下拉升级为可复用 `CitySelector` 组件。用户输入关键词后，组件优先展示城市候选。若某个城市仅对应一个站点，则点击后直接完成选择；若某个城市下存在多个站点，则在组件内部展开二级站点列表，用户必须进一步选择具体站点。

后端保留现有查询接口与站点校验逻辑，只新增城市候选能力。城市候选由现有 12306 站点列表聚合得到，作为选择阶段的视图模型存在，不进入核心查询模型。

## 架构设计
- 保持 `POST /api/search` 的请求结构不变，继续接收 `departure_station` 与 `arrival_station`
- 在站点数据之上新增城市聚合层，生成城市候选集合
- 前端新增可复用 `CitySelector` 组件，替代当前字段级站点下拉控制器
- 页面保留两个输入位：出发城市、到达城市；但最终隐藏字段始终保存具体站点名

## 前端组件设计
### 组件职责
`CitySelector` 负责以下能力：
- 管理输入框值、隐藏站点值、候选列表和展开状态
- 根据关键词请求城市候选
- 渲染单站城市和多站城市两类结果
- 在多站城市场景下展开并确认具体站点
- 处理键盘导航、点击选择、失焦关闭、错误提示

### 组件状态
组件至少维护以下状态：
- `query`: 当前输入框内容
- `selectedStation`: 当前已选具体站点
- `items`: 当前城市候选结果
- `activeIndex`: 当前高亮项
- `expandedCity`: 当前已展开的多站城市
- `requestId`: 用于丢弃过期请求
- `loading`: 是否正在加载候选
- `error`: 最近一次候选请求是否失败

### 交互规则
- 用户输入中文、拼音或简拼后，请求城市候选
- 单站城市：点击或回车后直接选中，写入隐藏站点值
- 多站城市：首次选择只展开其站点列表，不立即写入隐藏值
- 用户在二级列表中选中具体站点后，才写入隐藏值并完成组件选择
- 输入框可显示城市名；组件下方显示“已选站点：xxx”以说明最终提交值
- 用户修改输入内容时，若与已选结果不再一致，立即清空隐藏站点值
- 无候选时显示空状态文案
- 请求失败时显示轻量错误提示与可重试状态

### 键盘交互
- `ArrowDown` / `ArrowUp`：在当前可见候选中移动高亮
- `Enter`：
  - 若高亮项是单站城市，直接确认
  - 若高亮项是多站城市，展开其二级站点列表
  - 若高亮项是二级站点，确认该站点
- `Escape`：关闭候选面板

## 模板与脚本调整
- `app/templates/index.html` 保持表单结构稳定，继续保留出发/到达两个字段位
- 两个字段继续使用隐藏输入保存最终站点名
- `app/static/app.js` 中当前的 `createStationField` 升级为更清晰的 `CitySelector` 组件实现
- 组件应以可复用方式初始化两个字段，避免复制粘贴逻辑

## 后端服务设计
### 数据模型
在现有 `StationOption` 之上引入城市视图模型 `CityOption`，字段建议如下：
- `city_name`: 城市名
- `matched_by`: 命中方式，如 `name` / `pinyin` / `abbr`
- `stations`: 该城市下的站点列表
- `display_label`: 前端用于展示的补充文本，例如“4 个车站”

### 聚合规则
首版采用显式且可解释的归并策略：
- 如果站点名就是城市名，则归为该城市
- 如果站点名以城市名为前缀，且后缀属于常见站点后缀（如南、西、东、北、朝阳等站位后缀），归入该城市
- 避免做难以解释的模糊智能归并，优先保证可控性和可验证性
- 若某些站点无法稳定归并，则保留为单站城市候选

### 接口设计
新增接口：`GET /api/cities?q=...&limit=...`

返回结构示例：

```json
{
  "cities": [
    {
      "city_name": "北京",
      "matched_by": "name",
      "display_label": "4 个车站",
      "stations": [
        {"name": "北京", "telecode": "BJP", "pinyin": "beijing", "abbr": "bj"},
        {"name": "北京南", "telecode": "VNP", "pinyin": "beijingnan", "abbr": "bjn"}
      ]
    }
  ]
}
```

保留现有 `has_station()` 与 `search_trips()` 的语义不变。城市只用于选择过程，不进入最终业务查询参数。

## 错误处理
- 候选接口失败时，组件展示轻量错误提示，不阻断页面其他操作
- 用户只选中城市但未确认具体站点时，不允许提交，并提示“请选择具体车站”
- 用户手动修改输入值后，如果与当前已确认站点不一致，隐藏值立即清空
- 如果后端最终校验发现站点无效，继续沿用现有错误返回方式
