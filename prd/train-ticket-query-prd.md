# PRD：指定日期 A 城市到 B 城市火车票查询与候补推荐

## 1. 文档目标

定义一个面向购票场景的查询与推荐能力：
在用户输入指定出发日期、出发城市 A、到达城市 B 后，系统不仅查询常规直达车票，还要进一步考虑：

- 直达
- 买长坐短
- 转车

最终输出三种分类下有现票的侯选项，并给出全局最优选项。

---

## 2. 背景与问题

用户在热门线路上经常遇到以下问题：

- 直达车次无票或仅剩不合适席别
- 实际可以通过“买长坐短”获得可上车区间，但用户难以自行发现
- 实际可以通过一次转车完成出行，但人工筛选成本高
- 某些车次存在跨天运行，导致买长坐短的终点、到站日期、换乘衔接判断变复杂

因此需要一个统一的查询能力，在常规购票路径之外，自动补充可执行的候补方案，并给出推荐排序。

---

## 3. 产品目标

### 3.1 核心目标

当用户查询“指定日期 A 城市到 B 城市”的火车票时，系统应：

- 返回三种分类下的候补方案
- 每种分类都给出可执行、可解释的候补选项
- 在全部候补方案中选出一个“最优选项”
- 正确处理买长坐短场景中的跨天车次

### 3.2 成功标准

- 用户无需手动逐车次分析，即可看到可候补的替代方案
- 用户能清楚理解每个方案属于哪一类、为什么可行、代价是什么
- 对跨天车次的时间与到站逻辑判断准确，无明显错判

---

## 4. 适用用户

- 查询热门线路的普通购票用户
- 接受替代购票方案的用户
- 对时间较敏感、希望快速得到最优建议的用户

---

## 5. 需求范围

### 5.1 本期范围

支持以下能力：

- 输入指定出发日期、A 城市、B 城市
- 查询 A 到 B 的直达候补方案
- 查询“买长坐短”候补方案
- 查询“一次转车”候补方案
- 输出三类候补方案及全局最优选项
- 处理跨天车次的日期计算与展示

### 5.2 暂不包含

- 两次及以上转车
- 抢票自动下单
- 多乘客联合最优拆分
- 跨站换乘中的城市内交通规划
- 价格预测、放票预测

---

## 6. 核心概念定义

### 6.1 指定日期

指用户期望从 A 城市出发的日期，记为 `D`。

### 6.2 候补选项

指当前用户目标区间无可直接购买余票时，系统识别出的可提交候补或可作为候补替代方案的车票方案。

### 6.3 三种分类

系统返回的候补方案按如下三类组织：

1. **直达候补**  
   同一车次直接从 A 到 B，无需延长购票区间，也无需转车。

2. **买长坐短候补**  
   用户实际上从 A 乘车到 B，但购票区间为 `A -> C` 或更长区间，其中 `C` 为 B 之后的后续停靠站。用户在 B 下车，不乘坐后续区间。

3. **转车候补**  
   用户通过两段车次完成 `A -> X -> B` 的行程，其中 X 为中转城市或中转站。

### 6.4 最优选项

指系统基于既定排序规则，从全部候补选项中计算出的单一推荐结果。

---

## 7. 用户故事

### 7.1 直达无票时寻找最简单替代方案

- 作为用户，我希望在 A 到 B 无票时，系统优先告诉我是否还有直达候补可选。

### 7.2 接受买长坐短方案

- 作为用户，我希望系统自动告诉我哪些“买到更远站”的方案可以覆盖我的实际行程。
- 作为用户，我希望系统明确标识额外多买了哪一段、增加了多少成本。

### 7.3 接受一次转车方案

- 作为用户，我希望系统告诉我是否存在合理的一次中转候补路径。
- 作为用户，我希望系统展示换乘等待时间、总时长和风险。

### 7.4 跨天场景下仍得到正确结果

- 作为用户，我希望夜间列车、次日到达、长区间跨天等场景下，系统仍能正确判断方案是否成立。

---

## 8. 输入与输出

### 8.1 输入参数

必填：

- 出发日期 `D`
- 出发城市 `A`
- 到达城市 `B`

可选扩展参数（建议预留）：

- 席别偏好（如二等座、一等座、硬卧等）
- 出发时间范围
- 是否接受转车
- 是否接受买长坐短
- 最大换乘等待时长
- 最大额外票价溢价

### 8.2 输出结果

系统需要返回：

- `直达候补` 列表
- `买长坐短候补` 列表
- `转车候补` 列表
- `最优选项`

每个候补选项建议包含：

- 方案类型
- 涉及车次
- 购票区间
- 实际乘坐区间
- 出发站、到达站
- 出发时间、到达时间
- 是否跨天
- 席别
- 票价
- 相对原始目标的额外成本
- 总耗时
- 候补可行性说明
- 推荐分

转车方案额外包含：

- 中转站/中转城市
- 第一段到达时间
- 第二段出发时间
- 换乘等待时长

---

## 9. 详细业务规则

### 9.1 直达候补规则

满足以下条件即可归入直达候补：

- 存在同一车次覆盖 `A -> B`
- 该车次在用户指定日期 `D` 从 A 发车
- 当前 `A -> B` 无可直接购买余票，但可提交候补，或业务上认定为候补型机会

展示重点：

- 车次与席别
- 到达时间
- 候补状态
- 与最优选项的比较

### 9.2 买长坐短候补规则

满足以下条件即可归入买长坐短候补：

- 存在同一车次覆盖 `A -> B -> C`
- 用户实际乘坐区间为 `A -> B`
- 购票区间为 `A -> C`，其中 `C` 是 B 之后的后续停靠站
- `A -> B` 当前不可直接购买，但 `A -> C` 存在可候补机会
- 车次、席别、上车时间满足购票规则

展示重点：

- 实际乘坐区间：`A -> B`
- 购票区间：`A -> C`
- 多买区间：`B -> C`
- 多付金额
- 是否跨天

#### 9.2.1 买长坐短跨天处理

这是本需求的重点之一。

系统必须正确处理以下情况：

1. **列车从 A 出发后于次日到达 B**  
   例如：`D` 日晚出发，`D+1` 日早到 B。

2. **购票延长后的终点 C 位于次日或更晚日期**  
   即 `A -> C` 的票面到站日期可能晚于 `A -> B`。

3. **B 与 C 不在同一天到达**  
   用户在 `D+1` 日于 B 下车，而票面终点 C 可能在 `D+1` 晚些时候或 `D+2` 才到达。

为避免错判，系统在买长坐短场景中必须：

- 基于列车完整运行时刻判断 `A -> B -> C` 的真实先后关系
- 使用“完整日期时间”而非仅“时:分”比较站点顺序
- 明确标记：
  - 是否跨天到达 B
  - 是否跨天到达 C
  - 多买区间跨了几天

#### 9.2.2 买长坐短有效性约束

以下情况不得作为有效候补方案：

- C 不是 B 之后的停靠站
- 购票区间与实际乘车区间不连续
- 列车时刻显示会在到达 B 前就结束购票区间
- 由于跨天处理错误导致 `C` 在时间上早于 `B`

### 9.3 转车候补规则

满足以下条件即可归入转车候补：

- 存在两段行程：`A -> X` 与 `X -> B`
- 第一段与第二段在时间上可衔接
- 两段均在业务上可作为候补机会
- 中转站 X 合理且换乘时间满足阈值

建议基础约束：

- 第一段到达时间 < 第二段出发时间
- 最小换乘时间满足站内换乘要求
- 最大换乘等待时长不超过系统阈值
- 不允许时间倒挂或跨站不可达的伪换乘

展示重点：

- 两段车次信息
- 中转站
- 换乘等待时长
- 总票价
- 总耗时
- 换乘风险提示

---

## 10. 候补结果分类与展示要求

查询结果页必须以四个区域输出：

1. **最优选项**
2. **直达候补**
3. **买长坐短候补**
4. **转车候补**

### 10.1 最优选项展示要求

最优选项需要清晰说明：

- 属于哪一类方案
- 为什么被推荐为最优
- 相比其他方案的优势

推荐文案建议包含：

- 时间最优
- 成本最优
- 综合最优
- 风险最低

### 10.2 分类列表展示要求

每个分类下：

- 至少展示 Top N（如 3~10）个候补方案
- 若无结果，展示“暂无可用候补方案”
- 每个方案需有可比较字段：时间、价格、额外成本、是否跨天、是否转车

---

## 11. 排序与最优选项规则

### 11.1 默认排序思路

建议采用综合评分，优先考虑：

1. 是否直达
2. 总出行时长
3. 候补成功可行性
4. 额外成本
5. 换乘复杂度
6. 是否跨天过多

### 11.2 分类优先级建议

若无特殊用户偏好，默认可采用以下优先级：

- 优先级 1：直达候补
- 优先级 2：买长坐短候补
- 优先级 3：转车候补

但在以下情况下允许低优先级方案成为最优选项：

- 直达候补总耗时明显更差
- 买长坐短额外成本过高
- 转车方案显著更早到达且换乘风险低

### 11.3 推荐分建议

推荐分可由以下因素组成：

- 直达加分
- 总时长越短分越高
- 额外成本越低分越高
- 换乘次数越少分越高
- 跨天次数越少分越高
- 候补可行性越高分越高

系统最终按推荐分排序，并取第一名作为“最优选项”。

---

## 12. 关键边界场景

### 12.1 跨天直达

- `D` 日发车，`D+1` 日到达 B
- 应正常展示为直达候补，不因跨天误判

### 12.2 跨天买长坐短

- `D` 日从 A 出发
- `D+1` 日到 B
- `D+1` 或 `D+2` 才到 C
- 必须正确判断 `A -> C` 方案可覆盖 `A -> B`

### 12.3 深夜转车

- 第一段深夜到达 X
- 第二段次日凌晨发车
- 若换乘时间满足阈值，应视为有效转车方案

### 12.4 无直达、仅有替代方案

- 直达候补为空
- 买长坐短或转车存在结果
- 仍需输出全局最优选项

### 12.5 三类结果都为空

- 明确返回“暂无候补方案”
- 不生成伪最优选项

---

## 13. 功能验收标准

### 13.1 基础验收

- 输入 `D + A + B` 后可返回结果
- 结果按三类分组展示
- 每类结果结构一致、字段完整
- 能输出唯一的最优选项

### 13.2 买长坐短验收

- 能识别 `A -> B` 无票、`A -> C` 可候补的场景
- 能展示实际乘车区间与购票区间差异
- 能计算额外多买金额
- 能正确处理 B、C 的跨天到达日期

### 13.3 转车验收

- 能识别一次转车方案
- 能计算换乘等待时长
- 不输出时间倒挂的错误方案

### 13.4 结果推荐验收

- 所有候补方案均有推荐分或等价排序依据
- 最优选项来自真实候补结果集合
- 最优选项有清晰推荐理由

---

## 14. 非功能要求

- 查询结果应具备可解释性，用户能理解推荐原因
- 日期时间计算必须统一基于完整时间戳
- 跨天展示必须对用户清晰，例如“次日到达”“后日到达”
- 输出结构应便于前端分组展示与排序扩展

## 17. 结论

该需求的核心价值在于：

- 不只查“有没有票”，而是查“有哪些可执行候补方案”
- 把复杂的替代购票路径自动化输出给用户
- 尤其在买长坐短场景下，通过正确处理跨天逻辑，避免漏召回和错推荐

本期建议以“直达候补 + 买长坐短候补 + 一次转车候补 + 全局最优选项”为最小可交付范围。
